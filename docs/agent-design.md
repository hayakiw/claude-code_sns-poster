# X(Twitter) 投稿エージェント設計 — /loop × /goal パイプライン

## 0. 設計思想 — loop と goal の組み合わせ方

このシステムの2本柱は `/loop` と `/goal`。

- **`/loop`（前段を自動で回す）**: `Trend Scout → Draft Writer → Typefully キュー登録` までを
  定期実行で自動巡回する。要件「**予約まで自動**」がこの範囲。
- **`/goal`（達成判定ゲート）**: キューに溜まった下書きと現在のKPIをゴールに照らして判定し、
  人間承認へ送るべきものを選ぶ。「数を回す loop」と「基準で判定する goal」を直列に置くのが核。
- **アウターループ（学習還元）**: `Post Analyst → Trend Scout`。
  反応分析の学び（改善メモ）を次サイクルのネタ収集に反映し、ゴールへ寄せていく。

> Typefully を **キュー兼承認UI兼公開エンジン**として使うことで、
> 「予約まで自動・**公開だけ承認**」が自然に成立する（下書きは人が公開するまで出ない）。

## 2. ノード詳細

### 2.1 Trend Scout（ネタ収集・トレンド調査） 🟪
- **入力**: Goal定義、改善メモ（Post Analyst の学び）、外部（トレンド/競合/季節/ニュース）
- **処理**: ネタ候補を収集 → ゴール適合度・新鮮度・リスクでスコアリング
- **出力**: `state/idea_pool.json`（スコア付きネタ候補）
- アウターループの受け口：前サイクルの改善メモをここで方針に反映

### 2.2 Draft Writer（下書き・コピー作成） 🟪
- **入力**: 採用ネタ、ブランドボイス、改善メモ、過去投稿台帳（重複回避用）
- **処理**: 本文（ツイート/スレッド）生成、フック・CTA付与、過去投稿との重複を簡易チェック
- **出力**: 投稿テキスト（Typefully へ渡す）

### 2.3 Typefully キュー（承認待ち下書きに登録） 🟩
- Draft Writer の出力を **Typefully API で下書き（draft）として登録**。任意で予約時刻を付与。
- Typefully の下書きは **人が公開操作するまで X に出ない** → これが承認待ち状態の実体。
- ここまでが `/loop` の自動範囲（予約まで自動）。

### 2.4 /goal 達成判定 ⬜
- キューの下書きと現在KPIを **Goal に照らして判定**（基準は §10.3）：
  - 各下書きを `fit_score` で採点 → 閾値超の上位N件を承認へ、未満は保留
  - 直近KPI進捗で方針を補正（未達→実証済みフォーマット重視 / 達成→新形式を試す）
- 基準を満たさない下書きは承認に上げず保留（goal による絞り込み）。

### 2.5 公開承認（人間がレビューして公開） 🟧
- **要件「公開だけ承認」のゲート**。人間が Typefully 上で下書きをレビューし、
  **承認＝公開（or 予約確定）／ 修正 ／ 却下** を判断。
- 承認操作（＝Typefully で公開/スケジュール）まで X には出ない。
- 監査用に `state/approval_log.json` へ記録。

### 2.6 公開済み（X に投稿される） ⬜
- Typefully 経由で公開された状態。台帳は `PUBLISHED`。

### 2.7 Post Analyst（反応分析・改善メモ） 🟪
- **処理**: 公開後のメトリクスを取得 → KPI 帰属 → 勝ちパターン抽出（時間帯・形式・フック）
- **出力**:
  - `state/post_ledger.json` に実績追記（履歴記録）
  - `state/playbook.md` に**改善メモ**を更新 → 次サイクルの Trend Scout / Draft Writer が参照
- アウターループを閉じる役。

## 3. /loop と /goal の役割分担

```
 ┌─ /loop（定期自動巡回・予約まで自動）──────────────┐
 │  Trend Scout → Draft Writer → Typefully キュー登録  │
 └────────────────────────────────────────────────┘
                       ▼
   /goal 達成判定（ゴール基準で選別・KPI進捗で方針補正）
                       ▼
   公開承認（人間 = 公開だけ）→ 公開済み → Post Analyst
                       ▼
   改善メモを次の /loop 巡回（Trend Scout）へ還元
```

- `/loop`: 手数を出す自動エンジン。周期は数時間〜日次で設定。
- `/goal`: 基準で絞る判定。loop が量を作り、goal が質と方向を制御する。

## 4. ループのカデンス

| 段 | 周期 | 主体 |
|---|---|---|
| 収集→下書き→Typefully登録（/loop） | 日次（3案生成） | Scout / Writer |
| /goal 達成判定 | /loop 毎 ＋ 週次集計（上位1〜2件を承認へ） | システム |
| 公開承認 | 各下書きを人がレビュー時 | 人間 |
| 反応分析・改善メモ（Post Analyst） | 公開後24–48h ＋ 週次 | Analyst |

## 6. Typefully 連携メモ
- **登録**: Draft Writer → Typefully API で draft 作成（任意で `schedule-date`、スレッド分割）。
- **承認待ち**: Typefully の下書きは公開操作まで X に出ない＝承認ゲートそのもの。
- **公開**: 人間が Typefully UI でレビュー → 公開/スケジュール確定。
- **計測**: 公開後の反応を **Typefully Analytics API** から取得（impressions＋engagement、X専用で対象一致・追加コストなし）。X API は pay-per-use のため不使用（§10.4）。
- 台帳の `typefully_draft_id` で内部状態と Typefully を双方向に突合。

---

## 7. ガードレール / リスク管理
- **重複チェック（Draft Writer）**: 過去・予約済みとの類似度上限。X自動化規約の連投回避
- **/goal ゲート**: ゴール未適合は承認に上げない
- **人間ゲート**: 公開は必ず人の操作（Typefully）を経由
- **キルスイッチ**: 公開後ネガ反応が閾値超 → /loop 一時停止＋人間アラート
- **監査性**: 承認ログ・台帳 history に全遷移を記録

## 9. 段階導入ロードマップ
- **Phase 1**: Scout→Writer→Typefully 手動登録→人間公開。台帳/改善メモを回し始める
- **Phase 2**: `/loop` で前段を自動化、Typefully API 連携、/goal 達成判定を導入
- **Phase 3**: Post Analyst の自動計測＋改善メモ還元を完全自動化
- **Phase 4**: A/B最適化、炎上キルスイッチ、マルチアカウント

## 10. 決定事項（推奨デフォルト）

未決だった5点を以下で確定（数値は運用で調整可）。

### 10.1 北極星KPI
- **主KPI: エンゲージメント率 = engagement ÷ impressions**（Typefully が両値を提供→算出可）。
  - 理由: コンテンツ品質で直接動かせる先行指標で X のアルゴリズム評価軸。フォロワー数は遅行・リーチ依存の成果指標なので副KPIに置く。
- **副KPI**: インプレッション（リーチ）／ プロフクリック→フォロー転換 ／ フォロワー純増。
- **目標設定**: 現状値がないので最初の2週間を**ベースライン計測期間**（目標なし）とし、以後「ベースライン比 +50%」を四半期目標に。ニッチ運用の絶対値目安は 3〜5%。
  - → ユーザー確定が要るのはここだけ（現状値・狙うターゲット）。

### 10.2 /loop 周期と本数
- **周期: 日次（1日1回）**。Scout→Writer→Typefully登録を一巡。
- **本数: 1サイクル3案生成 → /goal が上位1〜2件を承認へ → 投稿1〜2件/日**。
- 理由: X の健全カデンス（量より質）、人間レビューを1日1回にまとめられる、過剰投稿で自動化規約に触れない。
- 拡張: 鮮度が重要なら Scout のみ6h毎に分離（Phase 3）。

### 10.3 /goal 達成判定の基準
- **fit_score (0–1)** = 0.4×ゴール寄与（勝ちパターン適合）+ 0.2×ボイス一致 + 0.2×新鮮度 + 0.2×(1 − リスク)。
- **閾値 0.6** 以上を承認候補とし、上位N（=その日の投稿目標数）を選抜。0.6未満は保留。
- **KPI集計**: /loop 毎にスナップショット + 週次集計。
- **explore/exploit**: 直近7日エンゲージメント率が目標帯を**下回る→実証済みフォーマット重視（exploit）**、**上回る→新形式を試行（explore）**。loop が量、goal が方向を制御。

### 10.4 反応メトリクスの取得元
- **Typefully Analytics API を主**に（impressions＋engagement を提供、X専用で対象一致、追加コストなし）。
- **X API は不使用**（2026/2 以降 pay-per-use・新規は無料枠なし、読み取り $0.005/件）。**公開も Typefully 経由**にして書き込み課金（$0.015〜、リンク付き $0.20/件）を回避。
- Typefully にない深い指標（例: ブックマーク）が要るときだけ X API をスポット利用。

### 10.5 承認通知と無応答時の挙動
- **チャネル: Slack**（環境に連携あり・即アクション）。/loop 毎に「/goal 通過下書き＋Typefully ディープリンク」をダイジェスト通知。
- **無応答時: 自動公開しない（HOLD）**。「公開だけ承認」の主旨を守り、鮮度切れは関連ウィンドウ経過で失効/繰越。時間依存ネタは予約枠接近時にリマインド。

> 残るユーザー確認は 10.1 のKPI現状値・目標値と、通知先 Slack チャンネルのみ。
```
